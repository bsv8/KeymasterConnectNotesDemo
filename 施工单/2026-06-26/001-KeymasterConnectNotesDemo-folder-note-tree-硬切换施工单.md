# KeymasterConnectNotesDemo 文件夹树与笔记实体模型硬切换一次性施工单

## 1. 本单定位

本单不是补丁说明，不是分阶段迁移计划，不是“先兼容旧模型再慢慢切”的过渡文档。

本单定义一次**硬切换**：

- 废弃“note 以完整 path 为主真值、title 为附属字段”的旧模型
- 改为“文件夹实体 + 笔记实体”的显式树结构模型
- UI 上给用户的是“文件夹 / 文件名 / 右键菜单 / 拖拽移动”的文件系统感知
- 底层真值不再依赖 title 或 filename 直接参与 key 唯一性

后续实现、测试、验收以本单为单真值。若与旧施工单冲突，以本单为准。

## 2. 简述缘由

### 2.1 旧模型的问题

旧模型里 note 直接以完整 path 做主键，会天然带来这些问题：

- `title` 和 `path` 是两套名字，用户心智割裂
- 改标题容易退化成改 path
- move 与 rename 容易混在一起
- 文件夹只能从 path 推导，不是真实体
- 无法自然支撑右键菜单与拖拽
- 若 filename 直接参与 key，则名称会被 key 规则污染

### 2.2 新模型为什么更合理

本次确认的新方向是：

- 文件夹是显式实体
- 笔记是显式实体
- 笔记有稳定 `id`
- 文件夹有稳定 `id`
- 树结构由 `parentId` / `folderId` 派生
- 笔记顶部编辑的“标题”，在用户感知上就是“文件名”
- 但它不再承担底层唯一 key 角色

这样做的直接收益：

1. 笔记重命名只改 `title`
2. 笔记移动只改 `folderId`
3. 文件夹移动只改 `parentId`
4. 拖拽本质就是 move，不再额外发明第三套语义
5. title 可是完全自由字符串，不必受 key 规则污染

### 2.3 为什么必须硬切换

不能保留双轨：

- 旧 path 模型一套
- 新 folder/note 实体模型一套

否则后果非常明确：

- 右键菜单、拖拽、移动、删除都要写两套分支
- 存量数据兼容逻辑会把 demo 做脏
- “文件名是不是 key”会在代码里反复打架

因此本次要求：

- 不做兼容双写
- 不做过渡壳
- 不做“旧 notes 先还能读，后面再迁”
- 直接按新模型重写真值

## 3. 目标产品模型

页面仍然是单页 notes demo，但产品心智改成：

- 左侧：文件夹树 + 文件夹里的笔记
- 中间：笔记正文编辑区
- 顶部：当前笔记标题输入框，用户感知为“文件名”
- 右侧：标签、加密状态、时间信息、移动/删除动作

明确不要：

- `Workspace` 概念
- `inbox` 概念
- 用户手输完整 path
- 右侧再放一份 title 编辑

## 4. 最终真值模型

## 4.1 根节点

根目录 `/` 固定存在，且可选中。

但根目录不必作为一条真实 folder record 落库。

推荐固定语义：

- `parentId: null` 表示直属根目录的文件夹
- `folderId: null` 表示直属根目录的笔记

也就是说：

- 根目录是虚拟节点
- 不是普通可删实体
- 不是 `none` / “未选中” 的别名

补充定义：

- `selection` 必须显式区分：
  - `root`
  - `folder`
  - `note`
- 不能再用 `none` 代表根目录
- 左侧目录框架中，除具体文件夹节点、具体笔记节点、专用按钮/控件外的空白区域：
  - 点击 = 根目录选中
  - 右键 = 根目录选中并弹根目录菜单

根目录菜单至少包含：

- `新建文件`
- `新建文件夹`

明确不允许：

- 根目录只作为拖拽落点
- 根目录与“无选中状态”混用
- 根目录出现删除 / 重命名 / 移动菜单项

## 4.2 文件夹记录

```ts
interface StoredFolderRecord {
  v: 1;
  id: string;
  parentId: string | null;
  title: string;
  createdAt: number;
  updatedAt: number;
}
```

说明：

- `id` 用稳定 uuid
- `parentId` 为父文件夹 id；根下文件夹为 `null`
- `title` 是用户看到的文件夹名
- 不存 `ownerPublicKeyHex`
- 不存 `path`

## 4.3 笔记记录

```ts
interface StoredNoteRecord {
  v: 2;
  id: string;
  folderId: string | null;
  title: string;
  tags: string[];
  createdAt: number;
  updatedAt: number;
  cipher: {
    contentType: "keymaster.notes.markdown.v1";
    nonceBase64: string;
    cipherbytesBase64: string;
  };
}
```

说明：

- `id` 用稳定 uuid
- `folderId` 指向所属文件夹；根下笔记为 `null`
- `title` 在 UI 上表现为“文件名”
- `title` 不再承担 key 唯一性语义
- 不存 `ownerPublicKeyHex`
- markdown 正文仍然只以密文落库

## 4.4 存储容器

按当前登录身份分区存储，但分区信息由外层 storage key 提供，不写进 record。

推荐容器形态：

```ts
interface StoredNotesSpace {
  v: 1;
  folders: Record<string, StoredFolderRecord>;
  notes: Record<string, StoredNoteRecord>;
}
```

其中外层 localStorage key 仍可类似：

```txt
notes-demo:owner:{publicKeyHex}
```

但 `folders` / `notes` 记录本身不再重复写 owner。

## 4.5 唯一性规则

同一父文件夹下：

- 不允许两个子文件夹同名
- 不允许两个笔记同名

也就是说：

- `parentId + title` 在 folder 维度唯一
- `folderId + title` 在 note 维度唯一

禁止：

- 同目录重名文件夹
- 同目录重名笔记
- 静默覆盖
- 自动改名

## 5. title / 文件名语义

### 5.1 对用户

用户看到的是“文件名”。

当前笔记顶部有一个可编辑输入框，这个输入框就是文件名编辑入口。

### 5.2 对系统

系统内部字段继续叫 `title` 即可。

原因：

- 与编辑器顶部标题语义一致
- 不必把代码名强行改成 `filename`
- 但产品文案和 UI 应尽量叫“文件名”

### 5.3 不做 slug

本次明确：

- 不做 slug 化
- 不自动把中文/空格/符号改成 `my-note` 这类程序键
- title 就是自由字符串

但仍需保留少量底线约束：

- 不能为空
- trim 后不能为空
- 不允许纯空白

不再要求：

- 只能 `a-z0-9._-`
- 只能小写
- 必须可当 path segment

## 6. 标签规则

标签输入支持以下分隔符：

- 半角逗号 `,`
- 全角逗号 `，`
- 空格

写入前必须：

1. 先按上述分隔符分割
2. `trim`
3. 过滤空值
4. 转小写
5. 去重
6. 最终按 `string[]` 落库

建议保留现有限制：

- 每条 note 最多 24 个 tag
- 单 tag 最长 32 字符

## 7. connect 与加密边界

本次**不**改协议能力边界，仍然只接：

- `identity.get`
- `cipher.encrypt`
- `cipher.decrypt`

保存流程仍是：

1. 编辑器拿到 markdown
2. 点击保存
3. 调 `cipher.encrypt`
4. 写入 note record 的 `cipher`

打开流程仍是：

1. 选中 note
2. 调 `cipher.decrypt`
3. 解出 markdown
4. 灌回编辑器

明确不做：

- 明文 markdown 落 localStorage
- BlockNote JSON 落库
- 解密失败时覆盖旧密文

## 8. 左侧树与交互模型

左侧树必须同时展示：

- 文件夹节点
- 文件节点

### 8.1 选中规则

- 选中文件夹：右侧操作聚焦文件夹；新建笔记 / 新建文件夹都默认落在此文件夹下
- 选中笔记：中间打开该 note，顶部显示文件名输入框

### 8.2 根目录

- 根目录 `/` 可选中
- 在根目录下可新建文件夹
- 在根目录下可新建笔记

### 8.3 右键菜单

本次必须引入右键菜单概念。

#### 文件夹右键菜单

至少包含：

- `新建笔记`
- `新建文件夹`
- `移动`
- `删除`

#### 笔记右键菜单

至少包含：

- `移动`
- `删除`

### 8.4 拖拽

本次允许并推荐加入拖拽，但必须明确：

- 拖拽不是独立业务语义
- 拖拽只是 move 能力的另一种交互壳

必须支持：

- 笔记拖到文件夹上
- 文件夹拖到文件夹上
- 拖到根目录区域

明确不允许：

- 拖到笔记上
- 文件夹拖到自己内部
- 文件夹拖到自己的后代下面

## 9. 操作能力定义

## 9.1 新建文件夹

行为：

- 在当前选中文件夹下创建
- 若当前选中的是笔记，则在该笔记所属文件夹下创建
- 若当前选中为空，则默认在根目录下创建

写入：

- 新增一条 `StoredFolderRecord`

约束：

- 同父目录下 folder title 不可重名

## 9.2 新建笔记

行为：

- 在当前选中文件夹下创建
- 若当前选中的是笔记，则在该笔记所属文件夹下创建
- 若当前选中为空，则默认在根目录下创建

写入：

- 新建时先只创建前端内存态 draft
- 首次点击“加密保存”成功后，才真正生成 `StoredNoteRecord`
- 首次保存成功前：
  - 不落库
  - 不生成占位 note record
  - 不写空 `cipher`
  - 不持久化“未保存 note”

首次持久化时：

- 生成 note `id`
- 写入 `folderId`
- 写入 `title`
- 写入 `tags`
- 写入加密后的 `cipher`

也就是说：

- “新建 note”不等于“已创建持久化 note”
- “加密保存成功”才等于正式创建完成

## 9.3 重命名笔记

行为：

- 编辑区顶部改标题
- UI 文案表现为“文件名”

写入：

- 只改 `StoredNoteRecord.title`

明确不做：

- 改 note id
- 改 folderId
- 做 path rename

## 9.4 移动笔记

行为：

- 可从右键菜单触发
- 可从拖拽触发

写入：

- 只改 `StoredNoteRecord.folderId`

约束：

- 目标文件夹下若已存在同名 note，则阻断

## 9.5 删除笔记

行为：

- 右键菜单删除
- 可在右侧动作区删除

写入：

- 删除 note record

## 9.6 删除文件夹

行为：

- 右键菜单删除
- 点击后先检查是否为空

空的定义：

- 没有直接子文件夹
- 没有直接子笔记

处理：

- 若为空：允许删除
- 若不为空：弹提示框，阻断删除

明确不做：

- 递归强删
- 自动搬迁内部内容
- 自动清空后再删

## 9.7 移动文件夹

行为：

- 可由右键菜单触发
- 可由拖拽触发

写入：

- 改 `StoredFolderRecord.parentId`

约束：

- 不能移到自己下面
- 不能移到自己的后代下面
- 目标父目录下若已有同名文件夹，则阻断

## 10. 特殊情况处理

## 10.1 解密失败

处理保持不变：

- note 仍保留
- 不自动删除
- 不自动覆盖
- 顶部与正文进入不可编辑态
- 仍允许删除

## 10.2 删除非空文件夹

必须弹提示框。

提示框要明确告诉用户：

- 当前文件夹非空
- 首版不支持递归删除
- 需要先手动清空再删

## 10.3 笔记重名冲突

若目标文件夹下已有同名 note：

- 阻断
- 提示用户改文件名或改目标文件夹

## 10.4 文件夹重名冲突

若目标父目录下已有同名文件夹：

- 阻断
- 提示用户改文件夹名或改目标位置

## 10.5 拖拽非法目标

例如：

- 拖到笔记上
- 文件夹拖到自己内部
- 文件夹拖到自己的子孙

处理：

- 不执行移动
- 给出轻量错误提示

## 10.6 右键菜单与拖拽的关系

两者必须共用同一套 move / delete 真值逻辑。

明确不允许：

- 右键菜单走一套分支
- 拖拽走另一套分支

## 11. 不能怎么做

本次明确禁止：

1. 不能继续保留完整 path 为 note 主真值。
2. 不能再让 title 与 filename 分裂成两个用户概念。
3. 不能保留 `Workspace` / `inbox` 叙事。
4. 不能让用户手输完整 path。
5. 不能再用 path 规则限制笔记标题字符集。
6. 不能把 `folderPath` 再塞回 `StoredNoteRecord`。
7. 不能同时保留 `key path` 和 `folderId` 两套结构真值。
8. 不能在 folder / note record 里重复存 `ownerPublicKeyHex`。
9. 不能删除非空文件夹。
10. 不能自动覆盖同名笔记或同名文件夹。
11. 不能只做拖拽、不做右键菜单。
12. 不能只做右键菜单、不统一到底层 move 能力。
13. 不能在解密失败时允许重新保存覆盖密文。
14. 不能把标签作为原始字符串直接存，不拆分。
15. 不能在“新建 note”时提前落占位 record。
16. 不能把根目录继续实现成仅拖拽可达、不可选中的伪节点。

- 根目录菜单至少有新建文件、新建文件夹
- 文件夹菜单至少有新建、移动、删除
- 笔记菜单至少有移动、删除

## 14. 收口结论

本次硬切换后的最终模型固定为：

- 根目录：虚拟节点 `/`
- 文件夹：`id + parentId + title`
- 笔记：`id + folderId + title + tags + cipher`
- 文件名：UI 心智；底层字段是 note `title`
- move：统一底层能力；右键菜单与拖拽共用
- 标签：按 `,` / `，` / 空格 分割后存数组

如果后续实现时要裁剪范围，允许裁剪：

- 拖拽的动画细节
- 右键菜单的视觉复杂度
- 次要装饰交互

但不允许动以下核心边界：

- 不能回退到 note 完整 path 模型
- 不能恢复 `Workspace` / `inbox`
- 不能让 title 再承担底层 key 唯一性
- 不能把 `folderPath` 再塞回 note record
- 不能删非空文件夹
- 不能让右键菜单和拖拽各走一套真值逻辑
