# KeymasterConnectNotesDemo note 搜索结果页、文件树图标与展开状态持久化硬切换一次性施工单

## 1. 本单定位

本单不是“先给树加折叠，再慢慢改搜索”的分步方案，也不是在当前 sidebar 过滤逻辑上继续加条件分支的过渡补丁。

本单定义一次**硬切换**，把当前“搜索 / tag 过滤直接作用在左侧文件树”的实现，整体切到下面这套新定义：

- 左侧文件树永远展示真实的完整树；
- 搜索范围只包含 `note.title + note.tags`；
- 只要有搜索词或激活了 tag，右侧文档区切成“搜索结果页”；
- 搜索结果里只有 `note`，没有 `folder`；
- 结果项固定两行：
  - 第一行大字：文件名
  - 第二行小字：所在 path
- 点击结果后：
  - 打开对应 note
  - 左侧树自动从根展开到该 note 所在 folder
  - 选中该 note
- 左侧树的 folder 必须可开合；
- 用户手工开合后的树状态，刷新页面后仍能恢复；
- folder / note 前面必须改成明确图标，不再用当前的字符占位。

后续实现、联调、验收均以本单为单真值。若与当前代码、旧搜索逻辑或旧施工单的局部描述冲突，以本单为准。

## 2. 简述缘由

### 2.1 让搜索直接裁剪文件树，会把两个问题缠成一个问题

当前 sidebar 已经承担：

- 根目录 / folder / note 树显示
- 选中态
- 右键菜单
- 拖拽移动
- 移动模式高亮
- 窄屏开合

如果再让 `searchQuery + activeTag` 直接裁剪树，会继续引入下面这些复杂度：

1. 匹配到 note 但祖先 folder 没匹配时，树上要不要补祖先壳？
2. folder 折叠时，命中的 note 要不要强制展开？
3. 搜索态下拖拽目标不完整，用户看到的是“半棵树”还是“真实树”？
4. 搜索清空后，之前的展开状态该恢复哪一份？
5. tag 搜索和标题搜索叠加后，树上“看不见但实际上存在”的节点会越来越多。

这不是一个“多写几行 if”能稳住的问题，而是数据语义已经开始打架。

### 2.2 搜索结果页比“过滤树”更简单，也更符合当前项目的简化原则

本项目的设计倾向一直是：

- 优先系统简单；
- 不为了边缘完整度引入多套并行状态；
- 真有问题时，宁可硬切到更干净的定义，也不堆补丁。

因此本次最合理的收口方式不是继续修树过滤，而是：

- 树只负责树；
- 搜索只负责结果；
- 点击结果只是一次正常的“选中 note”；
- 额外补一个“把祖先路径展开出来”的动作。

这样每块逻辑都有单一职责，不需要再让 sidebar 一边做树、一边做搜索结果容器。

### 2.3 展开状态必须有单真值，而且必须可持久化

当前 folder 看起来前面有一个 `▾`，但实际上所有 folder 永远展开，并不存在真正的“展开状态”。

这会带来两个问题：

1. 用户看到的是假交互；
2. 后面一旦需要“点击搜索结果后展开祖先路径”，就没有可靠状态可写。

因此本次必须把“哪些 folder 处于展开态”上升为显式真值，并满足：

- 只有一份状态源；
- folder 开合和搜索结果自动展开都写这份真值；
- 刷新页面后可恢复；
- 切换 owner 时按 owner 隔离。

### 2.4 搜索结果项必须把标题和 path 分开，不要把信息揉成一行

你已经明确要求结果项是“两行”。

这不是视觉偏好问题，而是交互信息密度问题：

- 大字标题让用户第一眼看文件名；
- 小字 path 让用户在重名、相近标题、不同项目目录下快速区分。

如果仍然把标题和 path 压成一行，搜索命中一多就很难扫。

## 3. 本次硬切换的最终目标

本次完成后，系统必须达到以下状态：

1. 左侧文件树不再被 `searchQuery` 或 `activeTag` 过滤。
2. 搜索范围只包含 `note.title + note.tags`，不包含 `folder.title`，也不包含正文内容。
3. 只要 `searchQuery !== ""` 或 `activeTag !== null`，右侧文档区就切到搜索结果页。
4. 搜索结果里只出现 `note`。
5. 每个结果项固定两行：
   - 第一行：大字标题
   - 第二行：小字 path
6. 点击搜索结果后，会打开对应 note，并把左侧树从根展开到该 note 所在 folder。
7. 文件树里的 folder 前面显示文件夹图标；note 前面显示文档图标。
8. folder 节点支持真实开合，不再是静态字符假装可展开。
9. 用户手工开合后的 folder 状态会持久化；刷新页面后保持原样。
10. 搜索结果触发的“祖先路径自动展开”也会写回这份持久化展开状态。
11. 现有保存拦截、解密失败、移动模式、拖拽、窄屏 sidebar 开合等业务规则不回归。

## 4. 最终交互与状态定义

## 4.1 搜索输入与 tag 过滤的最终语义

sidebar 里的两个入口保留：

- 搜索输入框 `searchQuery`
- tag 过滤 `activeTag`

但它们的作用不再是“过滤树”，而是“驱动右侧搜索结果页”。

最终定义：

1. `searchQuery`：
   - 仅匹配 `note.title`
   - 规则为 `trim + lowerCase + includes`
2. `activeTag`：
   - 仅匹配 `note.tags`
   - 规则为大小写不敏感的精确 tag 命中
3. 两者同时存在时：
   - 结果取交集
4. `folder.title` 永远不参与搜索
5. note 正文 markdown 永远不参与搜索

## 4.2 何时进入“搜索结果页”

新增明确判定，例如：

- `isSearchMode = searchQuery.trim().length > 0 || activeTag !== null`

规则固定为：

1. `isSearchMode === true`：
   - 右侧文档区显示搜索结果页
   - 不显示当前 note / folder / root 的普通详情页
2. `isSearchMode === false`：
   - 右侧恢复当前正常的 note / folder / root 视图

这条规则必须硬切，不做“有搜索时上面显示结果，下面还继续显示编辑器”的双栏混搭。

## 4.3 搜索结果集合的最终定义

搜索结果来源必须基于 `viewSpace.notes`，而不是只基于 `space.notes`。

原因：

- 当前未保存新 note 也是当前工作区的一部分；
- 若它的标题或 tag 命中搜索，应当出现在结果页；
- 否则用户会看到“刚刚创建的 note 明明存在，却搜不到”。

因此结果集合定义为：

1. 遍历 `viewSpace.notes`
2. 按 `searchQuery + activeTag` 判定是否命中
3. 为每条命中的 note 生成结果项

## 4.4 搜索结果排序

本单不引入“相关性评分”系统，不做高亮命中词，也不做模糊排序。

最终排序采用：

- 按文件树自然顺序输出

也就是：

1. 先按当前 folder 树的显示顺序遍历
2. 再收集其中命中的 note
3. 根目录直属 note 也遵循树上的显示顺序

这样排序稳定、简单，而且和用户左侧看到的树顺序一致。

## 4.5 搜索结果项的最终样式与点击语义

每个结果项固定两行：

1. 第一行：
   - 显示 note 标题
   - 字号大于第二行
   - 标题为空时显示 `未命名 note`
2. 第二行：
   - 显示 note 所在 path
   - 使用弱化小字
   - 根目录下的 note 显示 `根目录`
   - 非根目录显示类似 `根目录 / 项目 / 方案`

点击结果项后：

1. 若目标就是当前 note：
   - 不重复打开
   - 但仍要确保祖先路径已展开
2. 若目标不是当前 note：
   - 走现有 `trySelect({ kind: "note", id })`
   - 由原有未保存拦截逻辑决定能否切换
3. 若切换最终成功：
   - 左侧树展开到该 note 所在 folder
   - 对应 note 高亮选中
4. 若切换被未保存拦截阻断：
   - 不改变当前 selection
   - 不提前写展开状态
   - 保持当前结果页与当前编辑状态一致

## 4.6 文件树开合的最终真值

新增显式展开状态，例如：

- `expandedFolderIds: Set<string>`

它必须满足：

1. `App.tsx` 持有真值；
2. `NotesSidebar.tsx` 只接收：
   - 哪些 folder 已展开
   - 点击开合按钮后的回调
3. folder 手工开合只修改这份真值；
4. 点击搜索结果后自动展开祖先路径，也只修改这份真值；
5. 不允许再存在第二套“临时 auto expanded”并行状态。

## 4.7 展开状态持久化规则

展开状态必须单独持久化，不能塞进 `StoredNotesSpace`。

原因：

- `StoredNotesSpace` 是业务真值；
- folder 是否展开只是 UI 偏好；
- 把 UI 偏好混进业务数据，只会污染 schema。

因此本单要求：

1. 单独使用一份 localStorage key
2. key 必须按 owner 分区
3. value 只存 `folderId[]`
4. 非法值、坏数据、已不存在的 folderId 在加载时静默丢弃

首次没有记录时，默认策略固定为：

- 所有 folder 默认展开

理由：

- 这最接近当前用户已经看到的行为；
- 能避免硬切换后第一次刷新突然“树全没了”的感知回归。

## 4.8 点击 folder 行与点击开合按钮的职责分离

为避免把“选中 folder”和“展开 / 折叠 folder”混成一个动作，本单要求：

1. 点击 folder 行：
   - 只负责选中 folder
   - 移动模式下仍按原语义当作目标点击
2. 点击 folder 前面的开合按钮：
   - 只负责展开 / 折叠
   - 不改变当前选中态

不能做成“点整行既选中又开合”，否则会让 folder 选中行为变得不稳定，也会把移动模式的点击目标语义搅乱。

## 4.9 文件树图标的最终定义

本单不新增第三方图标库。

最终方案：

1. folder：
   - 使用内联 SVG 文件夹图标
   - 展开态 / 折叠态允许有轻微差异
2. note：
   - 使用内联 SVG 文档图标
3. 展开箭头：
   - 单独作为开合按钮显示
   - 不再复用图标字符 `▾`

原因：

- 当前项目没有现成图标依赖；
- 只为这次改动引入新图库不值当；
- 内联 SVG 更稳定，也更容易控制尺寸与颜色。

## 5. 特殊情况与提前决策

## 5.1 当前有未保存修改时，点击搜索结果

必须沿用现有“保存并切换”拦截。

规则：

1. 当前 dirty note 存在时，点击别的结果项：
   - 走现有 `trySelect`
   - 弹保存并切换遮罩
2. 在用户未确认前：
   - 不提前展开目标 note 的祖先路径
   - 不改变当前选中态
3. 保存成功并完成切换后：
   - 再展开目标路径

不能为了“搜索结果点击看起来快一点”，先把树展开再说。那会制造“左边像切过去了，右边其实还没切”的半状态。

## 5.2 当前 note 正在 loading

若当前 note 正在解密，而用户在搜索结果里点击另一条 note：

1. 仍走现有 note 切换链路；
2. 旧 pending decrypt 仍按现有机制 cancel；
3. 只有真正完成切换后，才展开新目标的祖先路径。

不能在 loading 态额外分叉一套搜索切换逻辑。

## 5.3 当前 note 为 decryptFailed

若当前 note 为 `decryptFailed`：

1. 搜索结果点击其他 note 仍允许直接切换
2. 因为当前 note 没有未保存修改概念
3. 切换成功后展开新目标路径

这条规则必须与当前树点击保持一致，不再为“结果页点击”另做例外。

## 5.4 当前未保存新 note 的搜索行为

当前未保存新 note 若命中 `title + tags`，应出现在搜索结果中。

规则：

1. 结果页可显示该 note
2. path 依据它当前的 `folderId` 计算
3. 若点击它自己：
   - 视为 no-op
   - 仅确保祖先路径展开

不能把未保存新 note 从搜索结果里排除，否则用户对“当前工作区里正在编辑的文件为什么搜不到”会很困惑。

## 5.5 folder 被删除、重命名、移动后的展开状态

展开状态是 UI 偏好，但不能长期残留坏 id。

规则：

1. 加载展开状态时，要过滤掉已不存在的 folderId
2. 每次 `space.folders` 变化后，也要清理一遍无效 id
3. folder 被移动或重命名：
   - 只要 id 没变，展开状态继续保留
4. folder 被删除：
   - 从展开集合中移除

## 5.6 搜索结果为空

当 `isSearchMode === true` 但结果为空时：

1. 右侧显示“无结果”页面
2. 页面上应明确当前搜索条件来自：
   - 关键词
   - tag
3. 左侧树仍保持完整，不被清空

不能在“无结果”时退回 folder / root / note 普通详情页，否则用户会误判搜索没有生效。

## 5.7 窄屏下点击搜索结果

窄屏下点击结果项后：

1. 仍遵循当前 sidebar 自动收起规则
2. 若 sidebar 当前已展开，可在切换成功后自动收起
3. 若切换被未保存拦截阻断，则不强制收起 sidebar

原则仍然是：只有成功切换后，才做 UI 收口。

## 6. 不能怎么做

下面这些实现，本单明确禁止：

1. 继续让 `searchQuery` 或 `activeTag` 直接过滤左侧文件树。
2. 在结果页里混入 `folder` 结果。
3. 把 `folder.title` 也纳入搜索范围。
4. 顺手做正文全文搜索。
5. 为了“自动展开搜索结果”，再维护一份并行的临时展开状态。
6. 把 folder 是否展开写进 `StoredNotesSpace` 业务数据里。
7. 点击搜索结果时，先改左侧树状态，再等待右侧是否真的切换成功。
8. 点击 folder 整行时同时触发展开 / 折叠。
9. 继续用 `▾`、`·` 这类字符假装图标。
10. 为这次改动引入完整第三方图标库。

## 8. 最终验收清单

以下清单全部通过，才算本单完成：

1. 登录后，左侧文件树默认展开全部 folder；folder 前是文件夹图标，note 前是文档图标。
2. 点击某个 folder 前面的开合按钮，只会展开 / 折叠该 folder，不会改变当前选中态。
3. 点击 folder 行本身，只会选中文件夹，不会顺手展开 / 折叠。
4. 手工折叠几个 folder 后刷新页面，展开状态保持不变。
5. 切换到另一个 owner 登录后，看到的是该 owner 自己的展开状态，不串数据。
6. 在搜索框输入标题关键字时，左侧文件树不被过滤，右侧切到搜索结果页。
7. 点击某个 tag 时，左侧文件树不被过滤，右侧切到搜索结果页。
8. 同时存在关键词和 tag 时，结果只显示两者都命中的 note。
9. 搜索结果里没有任何 folder。
10. 搜索结果项固定两行显示：第一行标题更大，第二行 path 更小。
11. 根目录下的 note 结果项第二行显示 `根目录`。
12. 非根目录下的 note 结果项第二行显示完整祖先路径。
13. 点击搜索结果中的某条 persisted note，右侧成功打开该 note，左侧树自动展开到其所在 folder，并高亮该 note。
14. 点击搜索结果中的当前 note，不会重复打开，但若祖先 folder 原先折叠，会被展开出来。
15. 当前 note 有未保存修改时，点击别的搜索结果，会先弹“保存并切换”，不会提前改变左侧树展开状态。
16. 保存并切换成功后，再展开目标路径并切过去。
17. 当前 note 为 `decryptFailed` 时，点击别的结果可直接切换，且切换成功后展开目标路径。
18. 当前 note 为 `loading` 时，点击别的结果仍走现有 cancel + 切换链路，不出现旧 note 晚回写 UI。
19. 当前未保存新 note 若标题或 tag 命中搜索，会出现在结果页中。
20. 搜索无结果时，右侧显示明确的“无结果”状态，左侧树仍完整可见。
21. 清空关键词且取消 activeTag 后，右侧退出搜索结果页，恢复普通 note / folder / root 视图。
22. folder 被删除后，不会在刷新后残留无效展开状态。
23. 移动、右键菜单、窄屏 sidebar 手工开合、保存遮罩等现有功能不出现行为回归。

## 9. 结论

本单的核心不是“给当前树补点搜索小逻辑”，而是一次明确的边界重划：

- 树只负责树；
- 搜索只负责结果；
- 展开状态只保留一份真值；
- 结果点击只是正常的 note 选中，再补一个祖先路径展开动作。

这套定义比继续维护“搜索态半棵树”简单得多，也更符合当前项目优先系统简化、避免并行真值的原则。
