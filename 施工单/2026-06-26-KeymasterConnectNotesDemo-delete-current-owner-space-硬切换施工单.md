# KeymasterConnectNotesDemo 删除当前账号数据（当前 owner 本地空间）硬切换一次性施工单

## 1. 本单定位

本单不是补丁说明，不是“先加个删除按钮试试”的小修，也不是分阶段迁移方案。

本单定义一次**硬切换**：

- 已登录态新增“删除当前账号数据”能力；
- 真实删除对象定义为“当前 `ownerPublicKeyHex` 对应的本地 notes 空间”；
- 删除成功后立即清空当前工作区内存态，并退回 `LockScreen`；
- 不引入 owner 列表、回收站、软删除、恢复机制；
- 不改 folder / note 数据结构；
- 不把 localStorage 方案升级成“每 owner 一个独立数据库”。

后续实现、测试、验收都以本单为单真值。若与旧施工单冲突，以本单为准。

## 2. 简述缘由

### 2.1 现在缺的不是“owner 字段”，而是“整空间删除入口”

当前实现已经满足这两个前提：

- folder / note record 本身不存 `ownerPublicKeyHex`
- notes 数据按 `notes-demo:owner:{publicKeyHex}` 分区

也就是说，现在真正缺的不是“去掉 owner 字段”，而是：

- 缺一个显式的“删除当前 owner 本地空间”入口
- 缺一条从 UI 到持久层的一次性整块删除链路

### 2.2 为什么不能把这件事叫“删除账号”

这个 demo 没有自己的账号系统，也没有服务端账号真值。

因此它能删除的只有：

- 当前浏览器里
- 当前站点下
- 当前 `ownerPublicKeyHex`
- 对应的本地 notes 数据

它**不能**删除：

- Keymaster 身份本身
- provider 侧账号
- 链上身份
- 其它站点下的数据

如果 UI 直接写“删除账号”，会把“本地数据删除”和“身份注销/销号”混为一谈，语义是错的。

### 2.3 为什么只删当前 owner 空间更合理

当前产品模型是：

- 登录后进入某个 owner 的 notes 工作区
- 所有读写都只针对当前 owner 分区

那么删除能力最自然的落点也应该是：

- 只删除当前 owner 的整块空间

这样做的收益：

1. 持久层动作最简单，直接删一个 key。
2. 不需要递归遍历 folder / note。
3. 不需要处理“删到一半失败”的中间态。
4. 与现有 owner 分区模型完全一致，不新增第二套数据真值。

### 2.4 为什么不建议升级成“每 owner 一个独立 DB”

如果把现在的 localStorage key 分区，升级成：

- 每个 owner 一个独立 IndexedDB database，或
- 每个 owner 一套更复杂的持久层命名空间

会立即引入这些复杂度：

- 存储 API 从同步变异步
- open / upgrade / deleteDatabase 失败路径变多
- 页面初始化与删除流程都要多维护一层状态机

当前 demo 的目标是“简单、粗暴、边界清晰地跑通”，不是把本地存储系统做重。

因此本次结论很明确：

- 保持现有 owner-key 分区
- 只补整空间删除能力

### 2.5 为什么必须硬切换

这次不能做成两套口径并存：

- 一边说“删除账号”
- 一边实际上只删 localStorage

也不能做成多条删除路径：

- 页头一条
- 未登录态 owner 管理页一条
- 调试入口再来一条

本次必须一次性收口为：

- 已登录态唯一删除入口
- 唯一删除对象 = 当前 owner 本地空间
- 唯一成功后果 = 删除后退回 `LockScreen`

## 3. 最终产品模型

## 3.1 删除入口放在已登录态页头

删除动作放在 `ConnectStatus` 所在的已登录页头区域。

原因：

- 当前 owner 身份信息就在这里
- “重新登录 / 切换身份 / 删除当前数据”属于同一组会话级动作
- 不污染 sidebar、editor、inspector 的业务动作区

本次不在未登录态放任何删除入口。

## 3.2 按钮文案必须精确

主按钮文案不应直接写：

- 删除账号
- 注销账号
- 删除身份

推荐文案：

- `删除当前本地数据`

确认弹窗中再把语义写全：

- 删除当前 `publicKey` 对应的本地 notes 数据
- 不删除 Keymaster 身份本身

也就是说：

- 按钮短文案追求紧凑
- 弹窗长文案负责把边界说清楚

## 3.3 删除语义

本次删除的真值定义为：

```txt
remove localStorage["notes-demo:owner:{publicKeyHex}"]
```

删除对象包含该 owner 空间下全部：

- folders
- notes
- 密文
- 明文 metadata

删除对象不包含：

- 当前站点其它 owner 的数据
- 任何 provider 账号信息
- 浏览器里其它站点数据

## 3.4 删除成功后的页面行为

删除成功后必须立即：

1. 关闭当前 popup session
2. 清空 `identity`
3. 清空 `space`
4. 清空 `pendingDrafts`
5. 清空 `selection`
6. 清空 `draft`
7. 清空 `decrypt cache`
8. 清空右键菜单 / 拖拽 / move / 弹窗等瞬时态
9. 清空搜索与 tag 过滤
10. 回到 `LockScreen`

这不是“切换身份但保留数据”，而是“当前 owner 数据已被删除并退出工作区”。

## 4. 核心设计约束

## 4.1 删除必须只依赖当前 owner

删除 key 的唯一输入是：

- `identity.publicKeyHex`

不能把这些东西混进删除维度：

- `targetOrigin`
- `currentOrigin`
- provider 品牌
- 上次登录时间

原因很简单：

- 当前 notes 空间本来就只按 owner 分区
- 删除语义必须与存储语义保持一致

## 4.2 同 owner 跨 provider 的删除语义必须说清楚

如果两个 provider 返回同一个 `ownerPublicKeyHex`，它们命中的是同一份本地空间。

因此删除一次的效果就是：

- 这个 owner 在本浏览器当前站点下的本地 notes 空间被整体删除

不能误导用户以为：

- 只删除了某个 provider 入口下的视图

## 4.3 删除必须是原子语义，不做逐条清理

不允许实现为：

- 先删 note
- 再删 folder
- 中间任何一步失败就停住

这样会平白引入部分成功态。

本次正确做法是：

- 直接删除 owner 对应的整块容器 key

## 4.4 删除失败时不能假装成功

这是本次的硬约束。

如果本地存储删除失败：

- 不能先把 UI 退回 `LockScreen`
- 不能提示成功
- 不能默默吞错

否则用户会以为数据没了，但实际刷新或重新登录后数据还在。

正确行为是：

- 保持当前登录态与工作区不变
- 展示明确错误
- 不做半成功清理

## 5. 交互与状态机

## 5.1 触发条件

只有在：

- `identity !== null`
- 当前不处于 `isLoggingIn`

时，才允许触发删除当前本地数据。

未登录态没有该入口。

## 5.2 二次确认

点击删除按钮后，必须弹二次确认框。

确认框文案必须同时说明三件事：

1. 删除范围是“当前 publicKey 对应的本地 notes 数据”
2. 这是不可恢复的本地删除
3. 不删除 Keymaster 身份本身

推荐文案示意：

```txt
确认删除当前本地数据？
这会删除当前 publicKey 对应的全部本地 notes 数据，并立即退出当前工作区。
该操作只影响本浏览器当前站点下的数据，不会删除 Keymaster 身份本身。
```

## 5.3 脏编辑态处理

如果当前存在这些情况：

- draft 未保存
- `pendingDrafts` 中有尚未首次保存的新 note
- 当前编辑器 markdown 已修改但未保存

本次不额外叠加一层“先提示你有未保存改动”的前置弹窗。

原因：

- 删除当前 owner 空间本身就是更高优先级、更强语义的破坏性操作
- 再叠一层未保存确认，只会把交互做复杂

因此规则明确为：

- 一个删除确认框吃掉全部风险提示

## 5.4 删除执行顺序

推荐执行顺序：

1. 读取当前 `identity.publicKeyHex`
2. 调用持久层删除函数
3. 只有删除函数明确返回成功，才执行 UI / 内存态清空
4. 退回 `LockScreen`

不能反过来先清内存再赌存储一定删得掉。

## 5.5 删除成功后的提示

本次不强制要求做额外 toast。

因为删除成功后页面会直接退回 `LockScreen`，状态变化已经足够明显。

如需保留提示，最多允许：

- 在 `LockScreen` 上显示一条短暂成功信息

但这不是本次必做项。

## 6. 特殊情况与处理原则

## 6.1 当前 owner 本地空间本来就不存在

例如：

- 当前 owner 第一次登录
- 当前空间本来就是空树
- storage key 已经被手动清掉

此时删除应视为**成功**。

原因：

- 用户想达到的最终结果是“当前 owner 在本地不再有数据”
- 若本来就没有，目标已成立

因此行为应为：

- 删除函数返回成功
- 正常清空内存态并退回 `LockScreen`

## 6.2 当前有解密失败 note

即使存在：

- `decryptFailed`
- 当前 note 无法重新打开

仍然允许删除当前 owner 空间。

原因：

- 删除整空间不依赖正文解密
- 这正是用户清理异常数据的一条出口

## 6.3 当前 popup session 仍存在

删除成功前后都应主动关闭当前 session：

- 防止旧 popup 继续持有 in-flight request
- 防止删除后还有异步结果回流污染新页面状态

处理原则：

- 删除成功后必须像“切换身份”一样关闭 session 并清空相关状态

## 6.4 当前正在重新登录

如果 `isLoggingIn === true`：

- 删除按钮禁用

不做并发竞争处理。

原因：

- “重新登录”与“删除当前 owner 数据”同时发生没有业务价值
- 强行支持只会引入竞态

## 6.5 localStorage 删除失败

例如浏览器抛出异常时：

- 不退出工作区
- 不清空 identity
- 不清空 space
- 直接展示错误

推荐错误文案方向：

- `删除当前本地数据失败，请重试。`

底层错误可以继续 `console.error`，但用户侧必须有明确失败提示。

## 6.6 删除后再次登录同一 owner

如果删除成功后再次登录同一个 owner：

- 应看到空树
- 不应恢复任何旧 folder / note

这是本次最核心的回归验证项之一。

## 7. 文件级实施方案

## 7.1 `src/lib/storage.ts`

新增并收口“owner 空间删除”能力。

必须新增：

- `deleteOwnerSpace(ownerPublicKeyHex: string): boolean`

推荐实现：

- 内部调用 `removeStorage(storageKeyForOwner(ownerPublicKeyHex))`
- 删除成功返回 `true`
- 删除失败返回 `false`

同时新增：

- `removeStorage(key: string): boolean`

设计要求：

- 不抛异常到上层
- 内部记录 `console.error`
- 以布尔值让上层决定是否继续退回 `LockScreen`

## 7.2 `src/App.tsx`

新增“删除当前 owner 本地数据”的页面级动作。

必须补齐：

- 删除动作触发函数
- 删除确认弹窗内容
- 删除成功后的整套内存态清理
- 删除失败后的错误呈现

推荐新增函数：

- `handleDeleteCurrentOwnerData`

实现要求：

- 先检查 `identity`
- 调用 `deleteOwnerSpace(identity.publicKeyHex)`
- 成功后复用或内联“切换身份”那套清理路径
- 失败后只设置错误，不清空当前工作区

同时要避免代码分叉：

- 不要把“切换身份清理内存”和“删除成功后清理内存”写成两份几乎相同的逻辑

推荐做法：

- 抽一个统一的工作区退出清理函数
- `handleSwitchIdentity` 与删除成功路径共用

## 7.3 `src/components/ConnectStatus.tsx`

在已登录态动作区增加删除按钮。

必须新增：

- 新的 props 回调，例如 `onDeleteCurrentData`

按钮要求：

- 仅已登录态显示
- `isLoggingIn` 时禁用
- 样式上应明显弱于主操作、但仍体现危险性

文案要求：

- 按钮本体不要写“删除账号”
- 用更精确的“删除当前本地数据”或等价文案

## 7.4 `src/styles.css`

若现有样式不足以表达危险动作，则补充页头删除按钮样式。

要求：

- 不引入新的复杂视觉系统
- 基于现有 `secondary-button` 轻量扩展
- 删除按钮与“切换身份”在视觉上能区分，但不过度喧宾夺主

## 7.5 `README.md`

更新文档边界说明。

至少补充：

- 已登录态新增“删除当前本地数据”能力
- 该能力只删除当前 owner 的本地 notes 空间
- 不删除 Keymaster 身份本身
- 删除成功后退回 `LockScreen`

## 8. 明确不能怎么做

本次明确禁止以下实现：

- 按钮直接命名为“删除账号”，但实际只删 localStorage
- 在未登录态展示 owner 列表并允许删除历史 owner 数据
- 把删除分区改成 `targetOrigin + owner` 双因子
- 为删除功能顺手引入 IndexedDB / 多数据库方案
- 逐条遍历 note / folder 做递归删除
- 删除失败时仍然清空 UI 并退回 `LockScreen`
- 增加“回收站 / 撤销删除 / 恢复最近删除 owner”
- 顺手扩展成“批量清理所有 owner 数据”
- 把 note / folder 上的普通删除逻辑与“删除当前 owner 空间”混成一套

## 9. 最终验收清单

以下验收项全部通过，才算本单完成。

### 9.1 入口与文案

1. 未登录态 `LockScreen` 不显示任何“删除当前本地数据”入口。
2. 已登录态页头显示删除入口。
3. 删除按钮文案不是“删除账号 / 注销账号 / 删除身份”。
4. 删除确认弹窗明确说明“只删除当前 publicKey 对应的本地 notes 数据，不删除 Keymaster 身份本身”。

### 9.2 正常删除路径

1. 用 owner X 登录。
2. 创建 folder / note，并至少保存一条 note。
3. 点击“删除当前本地数据”并确认。
4. 页面应立即退回 `LockScreen`。
5. 重新登录 owner X。
6. 应看到空树，不应恢复任何刚才的数据。

### 9.3 空空间删除

1. 用一个没有历史数据的 owner Y 登录。
2. 不创建任何内容，直接执行删除。
3. 页面应正常退回 `LockScreen`。
4. 重新登录 owner Y，仍应为空树。

### 9.4 未保存内容删除

1. 用 owner X 登录。
2. 新建 note 但不保存，或修改已有 note 但不保存。
3. 执行删除并确认。
4. 页面应直接退回 `LockScreen`。
5. 再次登录 owner X，应为空树，不应残留未保存 draft 或半成品数据。

### 9.5 解密失败场景删除

1. 构造一条当前无法解密的 note。
2. 在解密失败态下执行“删除当前本地数据”。
3. 删除应允许进行。
4. 成功后退回 `LockScreen`。
5. 再次登录同 owner，应为空树。

### 9.6 跨 provider 同 owner

1. 用 provider A 登录 owner X，创建并保存数据。
2. 切换到 provider B，若返回同一个 owner X，应能看到同一份本地数据。
3. 在任一 provider 下执行“删除当前本地数据”。
4. 再次使用 provider A 或 B 登录 owner X，都应为空树。

### 9.7 切换身份路径不回归

1. 用 owner X 登录并创建数据。
2. 点击“切换身份 / 更换登录器”。
3. 页面应退回 `LockScreen`，但不删除本地数据。
4. 再次登录 owner X，应仍能看到原数据。

### 9.8 删除失败路径

1. 人工模拟 `removeStorage` 失败。
2. 执行“删除当前本地数据”。
3. 页面不应退回 `LockScreen`。
4. 当前工作区内容应仍保留。
5. UI 应显示明确错误。

### 9.9 状态清理完整性

1. 删除成功前，制造这些状态中的至少几项：
   - 选中某条 note
   - 打开右键菜单
   - 处于拖拽或 move 模式
   - 有搜索关键字
   - 有 active tag
   - 有 confirmDialog / pendingDialog
2. 删除成功后退回 `LockScreen`。
3. 再重新登录任意 owner，不应看到旧工作区状态回流。

## 10. 收口结论

本次迭代的真实目标不是“支持删除账号”，而是：

- 在不改变当前 owner 分区模型的前提下
- 为当前登录 owner 提供一个明确、原子、不会误导用户的本地整空间删除能力

这条能力一旦落地，本 demo 关于“切换身份”和“删除当前数据”的边界就会变得清楚：

- 切换身份：只退回登录壳，不删数据
- 删除当前本地数据：删当前 owner 空间，再退回登录壳

后续若再讨论“历史 owner 管理”“批量清理”“真正账号删除”，都应视为新的独立需求，不属于本单范围。
